# 官制與行政區 · 朝代 register 藍圖

> `/translation-glossary`（翻譯定名）新分頁：**官名／職務名／行政區名**。
> 命名骨架取自使用者與 AI 共同推演的「漢譯世界史」對照（使用者提供、逐項討論定調）。
> AI 未經使用者明確拍板的細項標【提】；使用者明確定調的標【核】。最終仍逐項可校。

## 1. 問題（使用者原話）

翻外國政權的官名／職務名／行政區名時，過去**過於重複、沒有層次感** —— 動不動
「總督／行省」，時代也對不上。要的是：**外國政權的官制與行政區跟漢字文化圈有大致的
「時代對應」＋「政體體例對應」**，按對應朝代的官制用字。中國的詞若不夠用，**可借
日／韓／越的漢字官職**（統監、留守、道…）。

## 2. 核心洞見

- 現行 5 領域表處理**專有名詞**（某人某城）；官名／行政區是**制度性普通名詞**，正因如此
  LLM 每次退回一小撮現代詞（總督／行省／省長），才顯得重複。
- **「總督／行省／副王」不是錯，是「明清 register」的詞** —— 只該用在對應明清的政權
  （奧斯曼、蒙兀兒、俄、近世殖民帝國），不該套到波斯／羅馬／亞述。這正是「重複」的根因。
- 對應原則＝**共時性**：按「社會發展階段＋政治氣質」，**不按日曆年代**（漢摩拉比 −1750
  卻對齊春秋齊桓，因兩者都是「霸業＋立法」階段）。
- 另一利器＝**層級**：帝國晚期把 `大區行臺→州→郡`（Prefecture→Diocese→Province）
  逐層對到魏晉三級制，讀者一眼看出權力金字塔 —— 這就是使用者要的「層次感」。

## 5. 資料架構

沿用 `/translation-glossary` 慣例。**新增一表 `official_titles`**（官名／職務／行政區
共表，`entity_type` 分流）；register 對應表放**純核心 taxonomy**（不進 DB，比照 `DOMAINS`）。

### 5.1 純核心（`scripts/glossary_naming.py`，test-first ✅）
- `ADMIN_REGISTERS`：8 桶 profile（key/label_zh/source/central/local/note）。
- `REGISTER_BY_POLITY`：政權→register（值可為 list，羅馬＝["漢制","魏晉制"]）。
- `registers_for_polity()` / `register_for_polity()`。
- `check_register_valid()`：flag register 非合法值（防打錯）。
- `check_register_matches_polity()`：flag 某列 register 不在其 polity 允許集合內。
- `DOMAINS` 已加 `offices`（→ `official_titles`）。
- 測試：`scripts/tests/test_offices_register.py`（21 綠）。

### 5.2 DB 表 `official_titles`（`database/glossary-offices-schema.sql`）
共同核心欄 ＋ `entity_type`(官職/行政區/封爵/機構)、`polity`、**`register`**、
**`admin_level`**(中央/一級/二級/三級/羈縻/特區 → 層次感)、`rank_tier`。
`name_variants` 專門**收「舊的扁平譯」**（總督；行省）好對照。unique=(name_english, polity)。

### 5.3 UI
`pages/translation-glossary/index.vue` 加 tab「官制與行政區」。篩選：`register`（八桶下拉）
＋ `entity_type` ＋ `admin_level`。列展開顯示該 register 的 central/local 詞庫，
一眼看出譯名屬哪朝官制。編輯模式同他表。

## 6. 工作流（見 SKILL）
1. 翻書前定書中主要政權 → 查 register。
2. 抽官名／區劃清單 → 逐項對照建議中譯（hand-curated，**不走 LLM 亂填**，比照 places/rulers）。
3. 定案進 `seed_glossary_offices.py`（純資料、`--dry`、register 自檢、on_conflict）。
4. `export_glossary_from_db.py` 同步進 ebook-translate cheat sheet。
